## 3. New Gaps Discovered

### 3.1 Margin loan — in scope or out?
Biggest gap. Design handoff treats margin as a first-class concept (dedicated tab, KPIs, what-if, Settings sub-section, Net Worth ledger row). No schema, no TODO item, no mention in master-spec. **Resolve before schema migration.** Two clean options:
- **In:** Add `margin_account` (single row: broker, balance_cad, limit_cad, rate_pct, call_threshold_pct) and include in leverage-ratio / D/E / net-worth formulas. Update `snapshots` with `margin_balance_cad`. Rename leverage screen tabs "HELOC | Margin". Add margin fields to Settings → Borrowing.
- **Out:** Strip margin from the design handoff. Leverage screen is HELOC-only. Defer margin to v1.
- **Recommendation:** **In.** The design was done with margin; the spec is behind the design, not the other way. Marginal cost of adding one single-row table + one snapshot column + parallel formulas is low and the user has actual margin exposure per the design's mock data (~CAD 150k HELOC + margin balance in the ratio calc).

### 3.2 Today's % change (`changePct`) — source not specified
Design shows "Today" column on Cockpit, Holdings, Watchlist, Login pre-auth. Requires `(current_price - prev_close) / prev_close`. The `prices` table has no `prev_close`. `yfinance` returns both via `.info` or `.history(period="2d")`.
- **Recommended resolution:** Add `prev_close REAL` to the `prices` table. Populated on each refresh. Cache alongside `price`.

### 3.3 FX conversion point — store both currencies, or convert at display?
Design's `Holding` stores `priceCAD` + optional `priceUSD`. Spec implies store-native-currency + convert-on-read. These drift quickly if the FX rate changes between a price refresh and a page render.
- **Recommended resolution:** Store native price only (`prices.price` in ticker-native currency; `holdings.currency` tells you which). Convert to CAD **at read time** using the current cached BOC rate. Single source of truth, no drift. Matches spec §3 / §4.

### 3.4 Ticker delisting / rename handling
Not specified. yfinance returns `None` or empty DataFrame for delisted tickers. Behavior needed: hide? mark dead? surface in Settings?
- **Recommended resolution:** On refresh, if a ticker errors 2 runs in a row, set `prices.stale = 1` and show a row-level "stale" badge in Holdings. Do not delete. User can re-map or remove manually.

### 3.5 Questrade CSV type: Activity vs Position?
Design Settings footer reads "Supported: Questrade **Activity** CSV". Activity CSV is a transactions log (buys/sells/dividends), not a position snapshot. data-pipeline.md §2.1 says "ACB from the export is authoritative" and extracts "quantity, average cost per share" — which is a **Position** CSV shape, not Activity.
- **Why it matters:** Activity CSV needs running-ACB math (tax-lot rollup); Position CSV is a direct snapshot. Code complexity differs by 10×.
- **Recommended resolution:** Lock on **Position CSV** (snapshot). Activity CSV is deferred. Fix the design footer string when specs are updated. Confirm which export Questrade actually offers when the user drops the file.

### 3.6 Import idempotency + history
Spec says "existing rows for the same `(account_id, ticker)` are replaced" → holdings table is idempotent ✓. But Settings → Imports design shows "File | Broker | Rows | **Imported** | Re-import" — implying an import history log is tracked. No such table in module-1.
- **Recommended resolution:** Add `imports` table: `id PK, filename TEXT, broker TEXT, rows INTEGER, imported_at TIMESTAMP, account_id FK`. Cheap and makes the Re-import UX sensible.

### 3.7 Snapshot cadence contradiction
module-1 §2 `snapshots.date DATE PK` (one row per day). data-pipeline.md §5.2 has `refresh.py` "write a row to snapshots" on every scheduler run (4h cadence → 4+ rows per day, PK collision).
- **Recommended resolution:** One of:
  - **Daily-only:** `UPSERT` on DATE PK — each refresh overwrites the same day's row; last run of the day wins.
  - **Per-run:** Change PK to `(date, run_ts)` — keeps history, grows faster.
- **Recommendation:** Daily UPSERT. v0.1 has no snapshot UI; keeping it minimal until a historical chart is built (deferred to v1).

### 3.8 Manual-entry validation rules — not specified
HELOC rate, mortgage rate, property value, etc. — no bounds.
- **Recommended resolution:** Validate only at UI submit: non-negative for $ amounts, 0–30% for rates, reject NaN. Don't add DB CHECK constraints (SQLite re-migration friction). Phase 2 concern.

### 3.9 First-run password bootstrap
No flow spec'd. On day 1, `settings` has no password hash.
- **Recommended resolution:** On first launch, if no hash row exists, prompt a one-screen "Set password" form; store hash and proceed. No env var required (cloud-first philosophy — user may not touch the shell).

### 3.10 Streamlit session + timeout mechanics
`st.session_state` is per-browser-session, resets on hard refresh. A "15-min session timeout" with a "60-second warning banner" and "Stay signed in" button is non-trivial in Streamlit.
- **Recommended resolution:** Use `st.session_state` + a `last_activity_ts` field updated on every rerun; check on each rerun; if `now - last_activity > timeout`, set `authed=False`. Banner at `timeout - 60s`. Good enough for v0.1; it's not a real cookie, but the browser-tab lifecycle matches the desired UX.

### 3.11 Stale-data banner threshold
Threshold not set. Reasonable default:
- **Recommended resolution:** `summary.json as_of` older than **6 hours** during market hours, or **24 hours** outside market hours → show banner. Not user-configurable in v0.1. Copy: "Last updated HH:MM local · data may be stale."

### 3.12 Public view (in-app) toggle
Design has a "Full view / Public view" toggle in the app that blurs local-only values. Not specified in master-spec or module-2. Useful for previewing what the cloud deploy would show — but is it a scope item?
- **Recommended resolution:** **Keep.** It's a cheap UX win (`publicView` boolean + CSS blur class) and validates the Path C split during development. Add to module-2 §2 top-nav spec.

### 3.13 Settings → Borrowing sub-sections
Design Settings has a "Borrowing" section with HELOC + Margin sub-sections including "Utilization warning threshold", "Broker call threshold", "Warning banner threshold". Not in module-2 §F.
- **Recommended resolution:** Fold into module-2 §F once margin decision in 3.1 is made.

### 3.14 Account label free-text vs enum
`accounts.label` is TEXT (free), `accounts.account_type` is enum. UI shows badges per type — labels aren't rendered on pills. OK, but: multi-broker same-type (e.g. "RRSP at Questrade" + "RRSP at IBKR") needs label to disambiguate in Holdings table. Design doesn't show that. Edge case for v0.1 (one broker).
